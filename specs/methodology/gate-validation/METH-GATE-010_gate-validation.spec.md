## Intent

Перед передачей атома следующей роли Оркестратор выполняет gate-валидацию — проверку чек-листа обязательных критериев. Если хотя бы один критерий не пройден, атом возвращается предыдущей роли на доработку.

## Domain Rules

- **DR-1:** Gate не может быть пропущен, даже для "простых" задач. _Нарушение:_ артефакт без гарантии качества.
- **DR-2:** Все пункты gate обязательны. Частичное прохождение = непрохождение. _Нарушение:_ техдолг.
- **DR-3:** При непрохождении gate атом возвращается предыдущей роли с указанием непройденных пунктов. _Нарушение:_ потеря обратной связи.

## AC (Gherkin)

```gherkin
Scenario: Gate A — переход от Аналитика к Разработчику (успех)
  Given атом заполнен Аналитиком
    And Intent содержит ясную формулировку бизнес-задачи
    And Domain Rules непротиворечивы и содержат описание нарушений
    And AC написаны в формате Gherkin с позитивными и негативными сценариями
    And все blocking Open Questions закрыты
  When Оркестратор проверяет чек-лист Gate A
  Then Gate A пройден
    And событие GatePassed зафиксировано
    And атом передаётся Разработчику

Scenario: Gate A — непрохождение (→ DR-2, DR-3)
  Given атом заполнен Аналитиком
    And AC отсутствуют или написаны не в формате Gherkin
  When Оркестратор проверяет чек-лист Gate A
  Then Gate A НЕ пройден
    And событие GateFailed зафиксировано с перечнем непройденных пунктов
    And атом возвращается Аналитику на доработку

Scenario: Gate B — переход от Разработчика к Тестировщику (успех)
  Given атом дополнен Разработчиком
    And Tech Spec заполнен и согласован с Domain Rules
    And Implementation Notes содержат trade-offs и dependencies
    And DMT описывает агрегаты, события и инварианты
  When Оркестратор проверяет чек-лист Gate B
  Then Gate B пройден
    And событие GatePassed зафиксировано
    And атом передаётся Тестировщику

Scenario: Gate B — непрохождение (→ DR-2, DR-3)
  Given атом дополнен Разработчиком
    And Tech Spec противоречит Domain Rules
  When Оркестратор проверяет чек-лист Gate B
  Then Gate B НЕ пройден
    And событие GateFailed зафиксировано с перечнем непройденных пунктов
    And атом возвращается Разработчику на доработку

Scenario: Gate C — финальная валидация Тестировщиком (успех)
  Given Тестировщик завершил верификацию
    And тест-кейсы покрывают все AC сценарии
    And verification.status = passed
  When Оркестратор проверяет чек-лист Gate C
  Then Gate C пройден
    And атом полностью завершён
    And событие GatePassed зафиксировано

Scenario: Попытка пропустить gate (negative, → DR-1)
  Given атом находится на этапе Аналитика
    And Аналитик объявил завершение
  When кто-либо пытается передать атом Разработчику без проверки Gate A
  Then передача запрещена
    And Оркестратор обязан выполнить проверку Gate A
```

## DMT

- **Aggregate:** Gate (validate, pass, fail)
- **Events:** GatePassed, GateFailed
- **Value Objects:** ChecklistItem (criterion, passed: boolean), GateResult (gate_id, items[], passed: boolean)
- **Invariants:** все пункты чек-листа обязательны, частичное прохождение невозможно

## Constraints

- **[MAND]** Gate обязателен — не существует процедуры обхода или исключения
- **[FULL]** Все пункты чек-листа должны быть пройдены — частичное прохождение эквивалентно провалу
- **[FEED]** При провале gate обязательна обратная связь с указанием конкретных непройденных пунктов

## Decision Log

- **DL-1:** Три gate (A, B, C) — по одному на каждый переход роли. _Причина:_ каждая роль имеет свой чек-лист качества.
- **DL-2:** Бинарный результат gate (passed / failed) без промежуточных состояний. _Причина:_ исключение субъективности, ясность решения.
- **DL-3:** Оркестратор как единственный исполнитель gate. _Причина:_ независимость проверки от автора артефакта.

## Tech Spec

Gate реализуется как чек-лист в формате checklist. Оркестратор (человек или AI-агент) проходит по каждому пункту и фиксирует результат. Чек-листы могут быть автоматизированы через CI/CD (линтеры для frontmatter, проверка наличия секций, валидация Gherkin-синтаксиса).

### Чек-лист Gate A (Аналитик → Разработчик)
- [ ] Intent заполнен и содержит бизнес-задачу
- [ ] Domain Rules непротиворечивы, каждый содержит описание нарушения
- [ ] AC в формате Gherkin, есть позитивные и негативные сценарии
- [ ] Все blocking OQ закрыты
- [ ] Frontmatter валиден

### Чек-лист Gate B (Разработчик → Тестировщик)
- [ ] Tech Spec заполнен и согласован с Domain Rules
- [ ] DMT описывает агрегаты, события, инварианты
- [ ] Implementation Notes содержат trade-offs и dependencies
- [ ] Новых blocking OQ не создано (или все закрыты)

### Чек-лист Gate C (Тестировщик → Завершение)
- [ ] Тест-кейсы покрывают все AC сценарии
- [ ] verification.status = passed
- [ ] Нет открытых blocking OQ

## Implementation Notes

- **Trade-offs:** обязательность gate замедляет поток, но предотвращает накопление техдолга и некачественных артефактов
- **Dependencies:** METH-PIPE-010 (пайплайн ролей), METH-ATOM-010 (жизненный цикл атома)
- **Limitations:** ручной gate зависит от дисциплины Оркестратора; автоматизация требует CI/CD инфраструктуры
