## Intent

Каждый атом проходит через три роли последовательно. Каждая роль производит свои артефакты и передаёт атом следующей роли через gate-валидацию. Оркестратор координирует переключение ролей и проверяет gates.

## Domain Rules

- **DR-1:** Роли выполняются строго последовательно: Аналитик → Разработчик → Тестировщик. _Нарушение:_ артефакт без контекста предыдущей роли.
- **DR-2:** Каждая роль читает ТОЛЬКО свои секции + общие (Intent, Open Questions). _Нарушение:_ размывание ответственности.
- **DR-3:** Роль не может изменять секции другой роли. При обнаружении проблемы — OQ и возврат. _Нарушение:_ потеря контрактности.
- **DR-4:** Переключение роли объявляется явно маркером [РОЛЬ: ...]. _Нарушение:_ неясно кто автор артефакта.

## AC (Gherkin)

```gherkin
Scenario: Аналитик завершает свою работу
  Given атом создан и находится в _draft/ или корне директории
    And Аналитик заполнил все свои секции (Intent, Domain Rules, AC, Open Questions)
  When Аналитик объявляет завершение работы маркером [РОЛЬ: Аналитик завершил]
  Then атом передаётся на Gate A
    And Оркестратор проверяет полноту секций Аналитика

Scenario: Разработчик получает атом после Gate A
  Given Gate A пройден успешно
  When Оркестратор переключает роль маркером [РОЛЬ: Разработчик]
  Then Разработчик читает Intent, Domain Rules, AC, Open Questions
    And Разработчик заполняет секции Tech Spec, Implementation Notes
    And Разработчик НЕ изменяет секции Аналитика

Scenario: Тестировщик получает атом после Gate B
  Given Gate B пройден успешно
  When Оркестратор переключает роль маркером [РОЛЬ: Тестировщик]
  Then Тестировщик читает AC, Tech Spec
    And Тестировщик создаёт тест-кейсы и обновляет verification.status
    And Тестировщик НЕ изменяет секции Аналитика и Разработчика

Scenario: Разработчик обнаруживает проблему в требованиях (→ DR-3)
  Given Разработчик работает над атомом
    And обнаружена неоднозначность в Domain Rules
  When Разработчик создаёт blocking OQ
  Then атом возвращается Аналитику на доработку
    And Оркестратор переключает роль маркером [РОЛЬ: Аналитик — доработка]

Scenario: Попытка пропустить роль (negative, → DR-1)
  Given атом находится на этапе Аналитика
  When кто-либо пытается передать атом сразу Тестировщику
  Then передача запрещена
    And атом должен пройти через Разработчика и Gate B
```

## DMT

- **Aggregate:** Pipeline (switchRole, produceArtifact, returnToPrevious)
- **Events:** RoleSwitched, ArtifactProduced
- **Value Objects:** Role (Analyst, Developer, Tester), Artifact (spec sections)
- **Invariants:** строгая последовательность ролей, изоляция секций, явное переключение

## Constraints

- **[ORD]** Порядок ролей фиксирован и не может быть изменён: Аналитик → Разработчик → Тестировщик
- **[ISO]** Каждая роль изолирована в своих секциях — запись только в свои, чтение общих + своих
- **[TRACE]** Каждое переключение роли фиксируется маркером для трассируемости

## Decision Log

- **DL-1:** Три роли вместо двух или четырёх. _Причина:_ минимальный набор для покрытия цикла требование → реализация → проверка.
- **DL-2:** Явные маркеры переключения ролей. _Причина:_ трассируемость в git history, ясность ответственности.
- **DL-3:** Возврат через OQ, а не прямое редактирование. _Причина:_ сохранение контрактности между ролями, история решений.

## Tech Spec

Пайплайн реализуется через конвенцию маркеров в коммитах и комментариях. Оркестратор может быть человеком или AI-агентом, контролирующим соблюдение последовательности. Секции атома физически разделены заголовками Markdown — каждая роль знает свои заголовки.

## Implementation Notes

- **Trade-offs:** строгая последовательность замедляет процесс, но гарантирует качество на каждом этапе
- **Dependencies:** METH-ATOM-010 (жизненный цикл атома), METH-GATE-010 (gate-валидация)
- **Limitations:** при работе одного человека во всех ролях дисциплина переключения требует сознательного усилия
