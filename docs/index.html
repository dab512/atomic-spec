The problem we solve

Requirements live everywhere.
Which means nowhere.

Most teams today
✗Requirements in Confluence, tasks in Jira, discussions in Slack — constant context switching
✗A new developer doesn't know why a particular decision was made
✗Tester doesn't know what exactly changed in the sprint — tests everything or nothing
✗PM changes a requirement — developer finds out at code review, tester — never
✗Manager can't see the real status: what's implemented, what's untested, what's outdated
With Atomic Spec
✓Everything in git: requirements, decisions, tests, history — one repository, one history
✓git log spec.md — full history: who, when, why, at whose initiative
✓git diff release/v1..release/v2 — exact list of scenario changes

How it works

Three rules. The entire methodology.

01
Atom = one file

Each requirement is a separate .spec.md. Contains intent, domain rules, Gherkin scenarios, and platform contract. Adding a requirement = creating a file.

02
Position = status

File in folder root — verified and current. In _draft/ — has open questions. In _deprecated/ — history. Status visible without opening the file.

03
Git = decision journal

Who changed it, when, why, at whose initiative — in the commit message. Diff between release tags — exact list of changes for each role.
